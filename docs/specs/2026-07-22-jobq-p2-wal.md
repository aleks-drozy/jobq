# jobq P2 — Write-Ahead Log & Crash Recovery

Status: ratified 2026-07-22. Synthesized from a multi-agent design panel
(code briefing + group-commit design survived; the panel's judging stage was
lost to a rate limit, so the final calls below were made by the maintainer
and the reasoning is recorded here).

## The contract, reduced to two rules

The only externally visible acknowledgements jobq makes are (1) the return
of `Enqueue` and (2) the return of `Ack`. Therefore:

1. **An enqueue must be durable before `Enqueue` returns** (under the strict
   policy). 2. **An acked job must never come back.** Everything else —
   attempt counts mid-flight, leases, delivery state — may be lost and
   re-derived at the cost of a duplicate delivery, which the at-least-once
   contract explicitly permits. Designs that persist leases buy a lower
   duplicate rate, not correctness.

## Restart semantics (the previously open question — settled)

**Leases are not persisted. A restart is modelled as the simultaneous expiry
of every outstanding lease.** On recovery, every job that was in flight
re-enters the ready queue through the existing `retire()` path, attempts
intact — or dead-letters if its budget is exhausted. Rationale: this is
exactly what a consumer already experiences when a lease times out, so
recovery introduces no new semantics; and any persisted-deadline scheme must
still handle clock skew across restarts, buying complexity for a weaker
guarantee than it appears to give.

## Architecture

**One queue-wide WAL, not one per topic.** Topic actors encode records and
append them to a shared buffer under a short mutex; a single committer
goroutine flushes batches (group commit): while one batch is in `fsync`,
arriving records accumulate into the next, so fsync cost divides across
concurrent producers instead of multiplying. Actors never block on I/O.
Only enqueue replies are durability-gated (rule 1); settles are logged but
replied to immediately — a lost ACK record means a duplicate, never a loss.
The DLQ handoff (`ADOPT`) is one record, so the move is atomic in the log
(rule from the briefing: the handoff is the only other place true loss was
possible).

## Record format

Little-endian. Frame: `u32 len | u32 crc | u8 type | body` where
`crc = crc32c(len ‖ type ‖ body)` — the CRC covers the length field so a
corrupted length is detected, not used to skip into garbage. Castagnoli
(hardware path on amd64/arm64), not IEEE.

Types: `TOPIC` (id↔name, once per topic), `ENQUEUE` (jobID, topicID,
maxAttempts, enqueuedAt, delay, payload — **payload is written exactly
once**), `DELIVER` (jobID, attempts), `RETRY` (jobID, attempts — covers both
nack and observed expiry), `ACK` (jobID), `ADOPT` (jobID, dstTopicID,
deadLetteredAttempts, at), `DROP` (jobID), `CLEANSHUT` (empty, synced by
Close). Timestamps as `i64 sec + u32 nsec`, never UnixNano (the zero
`time.Time` does not round-trip through UnixNano — measured).

Two deliberate simplifications versus the panel's performance design, both
trading a few bytes per record for implementation and audit simplicity:
records carry the raw 16-byte job ID rather than LSN-relative references
(so encoding needs no lock-held LSN assignment), and topic IDs are the only
interning. Write amplification for a 256-byte job's full lifecycle is ~1.4×;
the by-reference scheme's 1.26× was not worth the recovery complexity.

## Segments and recovery

Segments `wal/%016x.seg` named by base offset, 32-byte synced header
(magic, baseOffset, seq, CRC) because **Windows cannot fsync a directory**
(measured: Access is denied) — file content, never directory state, is the
source of truth. Default segment size 32 MiB. Records never straddle
segments.

Recovery: read segments in base-offset order, verify every frame CRC,
apply records to rebuild each topic's ready list and attempt counts; on the
first bad frame in the **last** segment, truncate there (torn tail — counted
and reported); a bad frame in any earlier segment is corruption and recovery
refuses to guess (fail loudly with the offset). In-flight state reconstructs
as: `DELIVER` without a later `RETRY`/`ACK`/`ADOPT`/`DROP` → lease treated
as expired → `retire()`. Recovery returns a report: records applied, bytes
truncated, jobs restored, jobs dead-lettered during replay.

## Fsync policies (the benchmark deliverable)

`SyncAlways` — enqueue replies gate on the group fsync that covers their
record; what a crash loses: nothing acknowledged. `SyncInterval(d)` (default
5 ms) — replies release after buffered write; a background tick fsyncs;
crash loses ≤ d of acknowledged enqueues, stated plainly. `SyncNever` — OS
page cache decides; crash loses an unbounded suffix. The P4 benchmark table
(throughput + p50/p99 latency per policy × payload size) is the honest cost
sheet for these choices.

## Checkpoints

`checkpoint/%016x.ckpt`: full-state snapshot (topics, ready jobs with
payloads, attempt counts) + coveredOffset + CRC, written via temp file +
rename; a stale or torn checkpoint is simply ignored in favour of an older
one plus more replay (safe on rename-without-dir-fsync Windows). After a
checkpoint syncs, segments entirely below coveredOffset are deleted.
Triggered by segment rotation, not a timer.

## Integration

`Open(dir, opts...) (*Queue, error)` replaces bare `New` for durable use
(`New()` stays: a nil WAL is the in-memory mode — all P1 tests keep
passing unchanged). Topic actors gain a `log func(rec) waiter` hook;
single-process exclusivity via an O_CREATE|O_EXCL lock file containing the
PID (flock does not exist on Windows).

## Crash harness (P2's proof)

`TestCrashRecovery` (guarded by `-short` skip): the test re-executes its own
binary as a child (`os.Getenv("JOBQ_CRASH_CHILD")` pattern) pointed at a
temp dir; the child enqueues/consumes under `SyncAlways` while printing
acked enqueue IDs and acked-complete job IDs to stdout (flushed); the parent
kills it (`Process.Kill` — no flush opportunity) at a random point, replays
the dir, and asserts: every enqueue the child acknowledged is present or
terminally accounted; every job the child acked complete is NOT redelivered;
duplicates are permitted and counted. Repeated N times with logged seeds.

## What this still loses or duplicates (honest list)

- `SyncInterval`/`SyncNever` lose the stated suffix of acknowledged work.
- Any crash duplicates in-flight deliveries (contract-permitted).
- A crash between a consumer's side effect and its `Ack` re-runs the side
  effect — consumers must key on `Job.ID` (documented since P1).
- Media corruption in non-tail segments halts recovery by design; there is
  no quorum to repair from on one node.
- The clock is wall time; delays jump with the wall clock across restarts.

## Implementation order (TDD)

1. `record.go` — encode/decode + CRC + torn-frame detection (fuzz decode).
2. `segment.go` — writer with rotation + header verify on open.
3. `wal.go` — group-commit committer, the three policies, CLEANSHUT.
4. `recover.go` — replay + report; property: replay(log(ops)) ≡ state(ops).
5. Topic/Queue integration behind the nil-WAL check; `Open()`.
6. `checkpoint.go` — snapshot, prune, recovery preference order.
7. Crash harness; then benchmarks (P4 pulls the table into the README).
